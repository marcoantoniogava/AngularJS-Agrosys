#1 - qual a diferença prática entre timeout e $scope.apply()? 

  R: $scope.$apply() → força o Angular a atualizar a tela
$timeout → agenda algo + já chama o ciclo do Angular automaticamente

    exemplo $apply()
setTimeout(function() {
    $scope.nome = "Nícolas";
    $scope.$apply();
}, 1000);

aqui você usa porque o setTimeout é JS puro, Angular não sabe que algo mudou

exemplo $timeout
$timeout(function() {
    $scope.nome = "Nícolas";
}, 1000);

aqui NÃO precisa de $apply()

---> como Angular funciona por trás

Angular usa um sistema chamado digest cycle
é um loop que fica rodando e perguntando:
"alguma variável mudou?"
"alguma variável mudou?"
"alguma variável mudou?"
  
Angular roda o digest quando você usa:
ng-click
$http
$timeout
$interval
eventos do Angular
nesses casos, você NÃO precisa fazer nada
  
problema: código externo ao Angular
exemplo:
setTimeout(...)
cordova.plugin.http(...)
eventListener(...)
Angular NÃO sabe que isso aconteceu

então o que acontece?
$scope.nome = "Nícolas";

valor muda no JS mas a tela NÃO atualiza

solução: $scope.$apply()
$scope.$apply(function() {
    $scope.nome = "Nícolas";
});

isso fala pro Angular:
"ei, roda o digest aí porque eu mexi em algo"
então por que $timeout resolve?

porque internamente ele faz isso:
executa função
↓
chama $apply automaticamente
↓
atualiza tela

resumo mental 
mudou variável dentro do Angular → funciona automático
mudou fora do Angular → precisa avisar ($apply)

o two-way binding só funciona quando o Angular está ciente da mudança

use $apply() quando:
estiver fora do Angular (cordova, plugin, setTimeout)
use $timeout quando:
quiser delay + integração com Angular
NÃO use nada quando:
já estiver dentro de Angular (ng-click, service, etc)

Angular só atualiza a tela quando ele sabe que algo mudou
se a mudança acontece fora dele, você precisa avisar usando $apply()
$timeout já faz isso automaticamente
